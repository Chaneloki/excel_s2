# excel_game_s2 專案總覽

本檔案是整個 s2 專案的**全局說明**，記錄世界觀/角色設定、玩法設計、技術選擇、
開發工作流程，以及開發時必須遵守的規則。之後每次討論有新決定，這份檔案要
同步更新，避免設定散落在對話紀錄裡找不到。

各零件資料夾下若有自己的README（例如 [2godot_project/README.md](2godot_project/README.md)），
是該零件的細部說明，本檔案則是專案層級的總覽，兩者不重複但互相對應。

**美術/UI風格的權威來源是 [0mockup](0mockup) 資料夾**，本檔案只整理重點，
細節規則一律以該資料夾內的文件為準。

---

## 1. 專案是什麼

延續 **v2**（Excel教學遊戲，視覺小說+教學）的精神，但這次是全新專案：
同一個世界觀、不同時間線、全新角色、更進階的Excel教學內容、全新的玩法
——把「教學Excel」包裝成「**偵探解謎器**」，玩家要用Excel函數在案件資料裡
找線索、破案。

v2專案本身**不受影響、不會被修改**，s2是獨立的新專案。

---

## 2. 世界觀設定

- 延續v2的**西幻魔法世界**底色：魔法是窮人翻身的稀缺資源（像一種限定的
  特權token），一般人知道魔法存在，但接觸不到。
- s2跟v2是**同一個世界，不同時間線**。
- s2所有主角，本質上跟v2的角色是**同一群靈魂轉世**，但是：
  - **身份、長相全部不同**
  - **劇情中完全不提及「轉世」這件事**，玩家不會被告知這個設定，這是
    給原作粉絲的隱藏連結，不是劇情主線。

## 3. 角色設定

### 女主角（host，最大主視角）—— 暫名「莉希雅」
- **22歲**，**貴族出身**，大學剛畢業，偵探所創辦人。
- 性格視覺：冷靜、聰明、優雅、有一點距離感。
- 不熱衷魔法，反而把自己的魔法拿去做了一台「**數據計算儀**」——
  這個設定讓「Excel解謎器」在劇情上有合理的奇幻包裝，不是現代科技穿越，
  魔法感主要來自計算儀本身，不是她一直在施法。
- 開了一間**偵探所**，開業已滿1個月，**目前零客戶**，正在打廣告招生意。
- 外觀基準：銀色長髮、紫晶色銳利眼睛、黑色長裙+哥德刺繡、綠色絲質
  領巾(cravat)、銀鏈與精緻偵探配件。是「貴族偵探」，不是冒險者或舞會
  貴族。
- 參考圖：[1char_setting/host_setting.png](1char_setting/host_setting.png)，
  完整美術規則見 [0mockup/character_style_guide_v0.1.md](0mockup/character_style_guide_v0.1.md)。

### 第1主角
- 在host零客戶、廣告招不到生意的低谷期出現，**成為host的第一個案件**
  的委託人/相關人。
- 案件1解決後，**成為偵探所的第一名成員**。
- 人設、代表色、代表物件待Chapter 1故事定稿後補上
  （見 character_style_guide 的「待補內容」章節）。

### 其餘主角
- 全部共 **4位主角**（含host），其餘2位之後陸續加入，**先聚焦host與
  第1主角、案件1**，其餘人設later再展開。

## 4. 玩法設計

### 4.1 整體結構
延續v2的視覺小說模式：BG + 角色立繪 + 對話框 + 點擊翻頁，之後可能加入
動畫/影片。劇情中穿插「**地圖走查**」（場景切換、可點擊熱點找線索/道具）
跟「**Excel解謎**」（用數據計算儀破解案件關鍵問題）。

### 4.2 Excel解謎器（數據計算儀）
- 外觀走「**真Excel視窗**」風格（欄號、列號、fx公式列），但**只支援
  案件設計時指定的函數**，本關可用公式要常駐顯示在畫面右側清單。
- 玩家打了沒支援的公式，**不會報技術性錯誤**，而是用偵探系統語氣拉回
  主線（例如「這份案件暫時不需要該計算方式」），維持劇情沉浸感，
  語氣規則見 ui_style_guide 第4節。
- 案件目標必須常駐顯示，不可被玩家關閉。
- 對應 [excel_teaching.pdf](excel_teaching.pdf) 的教學內容，函數跟解謎情境的對照：

  | 函數 | 解謎情境 |
  |---|---|
  | COUNTIF / COUNTIFS | 找出口供重複/缺席的嫌疑人 |
  | SUMIF / SUMIFS | 核對帳目，抓金額異常的人 |
  | VLOOKUP / XLOOKUP | 用ID查口供表、物品保管記錄 |
  | INDEX+MATCH | 跨表交叉比對 |
  | 日期函數(DATEDIF/WEEKDAY等) | 還原案發時間線，抓矛盾 |
  | 文本函數(LEFT/RIGHT/MID/FIND) | 拆解假身份/暗號 |

### 4.3 案件資料規則
- **乾淨資料**（指向答案的關鍵數值）：案件設計時**寫死、固定不變**。
- **裝飾性資料**（陪襯NPC名字、無關數字等）：用邏輯規則生成，增加
  「每次玩都不太一樣」的真實資料庫感。
- **生成時機**：**新遊戲開始時生成一次，存檔後固定**——同一個存檔裡
  資料不會變動，避免玩家中途離開解謎器再回來時搞混。
- **陷阱/干擾項**（視覺相似但邏輯無關的假線索）：
  - **案件1：不要陷阱**（教學向，先讓玩家熟悉怎麼打公式）。
  - **之後案件**：逐步加入陷阱，難度遞增，跟主角陸續加入同步。

## 5. 美術與技術風格（已定案方向）

### 5.1 核心方向
**貴族偵探 × 魔法資料學 × 半寫實 × 安靜懸疑**，UI要像主角自己設計的工具：
冷靜、精準、漂亮，但不炫耀魔法。明確避免：可愛anime、現代辦公軟件、
cyberpunk全息、華麗魔法少女、普通RPG道具欄。

### 5.2 配色（取代先前「金/酒紅/象牙白」的暫定方向，以mockup實際呈現為準）
- 深炭黑/深木色：主要UI底色、面板背景
- 銀色/古銅金：邊框、分隔線、貴族感細節
- 深綠：數據計算儀、選取狀態、重要互動提示
- 象牙白：文字、紙張、案件文件感
- 暗酒紅：少量點綴，用於警告/重要線索
- 完整規則（材質感、文字語氣、按鈕風格）見 ui_style_guide_v0.1.md 第2-5節

### 5.3 角色美術
**半寫實 2.5D game illustration**，西方魔法世界+貴族城市+偵探懸疑，
明確避免anime大眼/chibi/手遊抽卡式裝飾。階級（貴族/學者/商人/勞工）
要靠服裝材質與細節表現，魔法元素克制（小型晶石、細線發光，不要全身
發光或大型魔法陣）。完整規則見 character_style_guide_v0.1.md。

### 5.4 技術選擇
- **引擎：Godot 4.x**——脫離v2手刻HTML/DOM的限制，UI系統更完整，
  日後可輸出exe或Web。
- **Excel模擬器：自製簡化版**，不接真實公式引擎（不用Luckysheet等
  套件），每個函數各自寫`_parse_xxx()` + `_evaluate_xxx()`，方便獨立
  測試、不互相干擾。

## 6. 開發工作流程（零件先行）

> 教訓來自v2/c2：檔案/邏輯混在一起做，後期很難維護。s2嚴格採用
> **零件先行**：每個系統先獨立做、獨立測試，最後才整合，**一次只做一個
> 零件**，不同時推進多個系統。

目前規劃的零件（會隨專案進度持續更新狀態）：

| 零件 | 狀態 | 說明 |
|---|---|---|
| 美術/UI風格指南 | ✅ 已定稿v0.1 | 見 [0mockup](0mockup)，6大畫面mockup + UI/角色風格規則文件 |
| Excel解謎器 - COUNTIF原型 | ✅ 已完成 | `01_excel_simulator_step1_countif.*`，見 [2godot_project](2godot_project)，假資料測試，已驗證輸入/運算/錯誤提示流程 |
| Story Dialogue UI 零件 | 🔶 v0.1進行中 | `02_story_dialogue_ui_demo.*`，規格見 story_dialogue_ui_component_spec_v0.1.md，目前是暫時單檔prototype，之後拆成DialogueBox/NamePlate/CharacterSpriteLayer等子零件 |
| Excel解謎器 - SUMIF / VLOOKUP / INDEX+MATCH / 日期 / 文本函數 | ⏳ 未開始 | 沿用COUNTIF的`_parse_xxx`/`_evaluate_xxx`模式逐個擴充 |
| 地圖走查系統（場景切換+熱點+道具欄） | ⏳ 未開始 | 獨立場景測試，畫面規則見ui_style_guide第8節 |
| Save/Load UI | ⏳ 未開始 | 畫面規則見ui_style_guide第9節 |
| Settings UI | ⏳ 未開始 | 畫面規則見ui_style_guide第10節 |
| 案件資料結構（JSON/資料檔，跟UI解耦） | ⏳ 未開始 | 純資料，方便之後換案件不用動程式邏輯 |
| 第一章故事劇本 | ⏳ 未開始 | **先完成劇本，才開始做整合Demo** |
| 整合Demo（案件1） | ⏳ 待第一章故事完成 + 各零件完成後才開始 | 把以上零件組裝成第一個可玩案件，流程見story_dialogue_ui_component_spec第12節 |

### 開發順序原則
1. ~~先寫第一章劇情劇本~~ → 美術風格指南已先行定稿（v0.1），劇本仍待補。
2. 各零件依上表**逐一**獨立開發、獨立測試，每個零件做完先用demo驗證。
3. **所有零件完成 + 第一章故事完成後**，才整合成Demo——不提前整合。

## 7. 嚴格規則（Strict Rules）

1. **溝通語言**：一律使用**繁體中文**回覆與寫文件。
2. **v2專案不可修改**：s2是完全獨立的新專案，不動v2任何檔案。
3. **零件先行，一次只做一件事**：不同時開發多個系統，每個零件獨立完成
   並驗證可動後，才進下一個。
4. **原型階段不放真實案件資料**：劇情/案件確定前，所有測試資料用
   無意義占位文字（例如「苹果/香蕉」），避免案件設計跟到原型測試。
5. **轉世設定絕對不能在s2劇情中提及或暗示**，玩家不可被告知任何角色
   的前世身份。
6. **檔名規範**：
   - 所有檔名一律使用**小寫字母**，空白用**底線`_`**取代（例如
     `story_dialogue_ui_mockup.png`，禁止`UI_style_guide v0.1.md`這種
     大小寫混雜+空白的命名）。
   - 檔名必須清楚反映用途（禁止`main.gd`、`test`、`scene1`、`new_script`
     這種無意義命名）。
   - 例外：Godot要求固定檔名的設定檔（如`project.godot`）維持引擎規定
     的格式，不在此規範內。
7. **程式碼規範**：
   - 所有註解使用**繁體中文**，並依**邏輯執行順序**分區說明
     （設定 → 建構 → 互動處理 → 核心邏輯）。
   - 禁止不必要的硬編碼（魔法數字/顏色散落各處）；**結構性常數**
     （表格大小、假資料等原型本身的設計參數）可以是具名常數，
     但**版面數字與配色必須集中管理**成單一來源。
8. **每個零件資料夾要有自己的README**，說明檔案用途與測試方式
   （參考 [2godot_project/README.md](2godot_project/README.md)）。
9. **美術/UI風格一律以 [0mockup](0mockup) 資料夾的文件與mockup圖為準**，
   其他地方（包含本檔案）的描述只是摘要，衝突時以0mockup為主。
10. **資料隨機化規則**：裝飾性資料可邏輯生成、每次新遊戲生成一次後
    鎖定；破案關鍵資料永遠固定不變，不可隨機。
11. **Excel解謎器不支援的公式**：必須用劇情/偵探系統語氣提示玩家拉回
    主線，不可顯示技術性錯誤訊息或讓遊戲卡死。
12. **案件難度曲線**：陷阱/干擾項從案件1的「無陷阱」開始，之後逐步
    遞增，不可一開始就太刁鑽。
